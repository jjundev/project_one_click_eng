# 리팩토링 계획 비판적 검토 기준

아래 기준으로 리팩토링 계획을 검토하라. 문제가 있는 항목만 지적하라.

**검토 항목**

필요성 — "클린 코드", "best practice" 같은 추상적 이유가 아닌, 실제 문제(크래시·ANR·메모리 릭·빌드 시간·확장 불가)에 근거하는가?
범위 — 필요 이상으로 넓지 않은가? 관련 없는 개선을 끼워 넣고 있지 않은가?
동작 보존 — 기존 단위 테스트·UI 테스트가 변경 없이 통과하는가? 테스트가 없는 영역이면 테스트 추가가 선행되는가? AndroidManifest.xml 권한·Intent Filter 변경이 누락되어 있지 않은가?
과잉 설계 — 아직 필요 없는 추상화(불필요한 UseCase 계층, 과도한 DI 모듈 분리, 사용처가 하나인 인터페이스)를 도입하려 하지 않는가?
단계 분리 — 중간에 중단해도 빌드가 깨지지 않는가? PR 단위로 분할 가능한가? Gradle 모듈 구조 변경과 코드 변경이 같은 단계에 섞여 있지 않은가?
위험도 — 가장 위험한 변경을 스스로 식별하고 있는가? DB 마이그레이션·ProGuard/R8 규칙·멀티 모듈 의존성 변경처럼 롤백이 어려운 항목이 분리되어 있는가?
누락 — 영향받는 리소스(res/layout, res/values, navigation 그래프), build.gradle 의존성, 난독화 규칙, 환경별 설정(BuildConfig, flavor) 등 빠진 것이 없는가?
근거 정확성 — 인용하는 라이브러리·API·Jetpack 컴포넌트가 실제로 존재하고, 프로젝트의 minSdk/compileSdk/Gradle 버전에서 사용 가능한가? Deprecated된 API를 새로 도입하려 하지 않는가?
